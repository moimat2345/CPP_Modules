02

Stack vs Heap — le grand sujet

La question centrale de l'ex00 : où vit ton objet, et qui le détruit ? C'est la décision mémoire en C++.

📚

La Stack (pile)

  • ⚡Rapide, automatique. Zombie z("x");
  • 🧹Détruit tout seul à la fin du { } (scope)
  • ⏱️Durée de vie = limitée au bloc courant
  • ✅Aucune fuite possible — tu ne gères rien
🗃️

Le Heap (tas)

  • 🔑Manuel. new Zombie("x") → renvoie un pointeur
  • ♾️Survit au-delà de la fonction qui l'a créé
  • 🧨Tu DOIS faire delete — sinon fuite mémoire
  • 🎯À utiliser quand l'objet doit vivre plus longtemps
🧭

La règle de décision : l'objet doit-il survivre à la fonction qui le crée ? Oui → heap (new). Non → stack. C'est exactement newZombie (heap, on le renvoie) vs randomChump (stack, usage local).

03

new / delete (et les tableaux)

Le C++ remplace malloc/free (interdits ici) par new/delete, qui appellent le constructeur et le destructeur. Pour un tableau, l'ex01 demande une seule allocation : new Zombie[N].

zombieHorde.cpp
Zombie* zombieHorde( int N, std::string name )
{
    if (N <= 0)
        return NULL;

    Zombie* horde = new Zombie[N];   // UNE seule allocation pour N objets

    for (int i = 0; i < N; i++)
        horde[i].setName(name);
    return horde;                  // pointeur vers le 1er zombie
}
⚠️

Le piège classique : ce qui est alloué avec new[] se libère avec delete[] (avec les crochets !), pas delete. Sinon comportement indéfini. → delete [] horde;

04

Pointeur vs Référence

Le cœur du module (ex02). Une référence, c'est juste un autre nom (un alias) pour une variable qui existe déjà. La preuve : dans l'ex02, la variable, le pointeur et la référence partagent la même adresse.

CritèrePointeur  std::string*Référence  std::string&
peut être nul ?oui (NULL)non — toujours valide
doit être initialisé ?non (dangereux)oui — à la déclaration
peut « pointer » ailleurs ensuite ?oui — réassignablenon — lié à vie
syntaxe d'accèsptr->x / *ptrref.x (comme la variable)
on stocke…une adresseun alias
ex02/main.cpp
std::string  brain = "HI THIS IS BRAIN";
std::string* stringPTR = &brain;   // un pointeur : stocke l'ADRESSE de brain
std::string& stringREF = brain;    // une référence : un AUTRE NOM pour brain

// &brain == stringPTR == &stringREF  → la MÊME adresse
// brain  == *stringPTR == stringREF   → la MÊME valeur
05

Membre référence vs membre pointeur

L'ex03 te fait choisir. Deux humains quasi identiques, mais une différence de design qui dicte tout :

🗡️

HumanA → référence

  • •Toujours armé, reçoit l'arme au constructeur
  • •Ne change jamais d'arme, jamais sans arme
  • →Membre Weapon& _weapon; (jamais nul)
🛡️

HumanB → pointeur

  • •Peut ne pas avoir d'arme (au début)
  • •L'arme s'ajoute après via setWeapon()
  • →Membre Weapon* _weapon; (peut être NULL)
🧠

La leçon : « quelque chose peut être absent ou changer » → pointeur. « toujours présent, fixé à la construction » → référence. Le code raconte l'intention.

06

Pointeurs sur fonctions membres

L'ex05 t'oblige à éviter la « forêt de if/else ». La solution : un tableau de pointeurs sur fonctions membres. Syntaxe étrange mais logique :

Harl.cpp — complain()
std::string levels[4] = {"DEBUG", "INFO", "WARNING", "ERROR"};

// un pointeur sur fonction membre de Harl, ne prenant rien, renvoyant void :
void (Harl::*funcs[4])( void ) = {
    &Harl::debug, &Harl::info, &Harl::warning, &Harl::error
};

for (int i = 0; i < 4; i++)
    if (levels[i] == level)
        (this->*funcs[i])();   // ← on appelle via le pointeur

À décoder : void (Harl::*funcs[4])(void) = « un tableau de 4 pointeurs vers des méthodes de Harl qui prennent void et renvoient void ». On les appelle avec (this->*funcs[i])().

07

Le switch et son fall-through

L'ex06 exploite un comportement souvent vu comme un bug : sans break, un case tombe dans le suivant. Ici c'est voulu : « ce niveau et au-dessus ». Clique un niveau 👇

Simulateur de ./harlFilter <LEVEL> — regarde quels messages s'allument selon le niveau choisi.

[ DEBUG ]I love having extra bacon for my 7XL burger...
[ INFO ]I cannot believe adding extra bacon costs more money...
[ WARNING ]I think I deserve to have some extra bacon for free...
[ ERROR ]This is unacceptable! I want to speak to the manager now.
[ Probably complaining about insignificant problems ]
ex06/main.cpp
switch (getLevel(argv[1]))
{
    case 0: harl.debug();   // pas de break → tombe dans le case suivant
    case 1: harl.info();
    case 2: harl.warning();
    case 3: harl.error(); break;
    default: std::cout << "[ Probably complaining... ]\n";
}
08

Les flux de fichiers C++

L'ex04 t'interdit les fonctions C (fopen, read…) et std::string::replace. Tu utilises les streams : std::ifstream (lire), std::ofstream (écrire), std::stringstream (tout charger en mémoire), et tu remplaces à la main avec find + substr.

FileReplacer.cpp — l'idée
std::ifstream  inFile(_filename.c_str());   // ouvrir en lecture
if (!inFile.is_open()) /* gérer l'erreur */;

std::stringstream buffer;
buffer << inFile.rdbuf();                // avaler tout le fichier d'un coup
std::string content = buffer.str();

// remplacer s1 par s2 SANS std::string::replace :
while ((found = content.find(_s1, pos)) != std::string::npos) {
    result += content.substr(pos, found - pos);
    result += _s2;
    pos = found + _s1.length();
}

📌 À retenir (défense d'éval)

  • →Stack = auto, détruit à la fin du scope. Heap (new) = manuel, survit, à delete.
  • →new ↔ delete, new[] ↔ delete[]. Jamais de mélange, jamais d'oubli (= fuite).
  • →Une référence = un alias : jamais nulle, init obligatoire, non réassignable. Un pointeur = une adresse : peut être nul et changer.
  • →Membre référence = « toujours là » (HumanA). Membre pointeur = « peut manquer/changer » (HumanB).
  • →Pointeur sur fonction membre : void (Harl::*f)(void), appel (this->*f)() → remplace les if/else.
  • →switch sans break = fall-through (volontaire dans l'ex06 pour « ce niveau et au-dessus »).
  • →I/O en C++ = streams (ifstream/ofstream), pas les fonctions C.
