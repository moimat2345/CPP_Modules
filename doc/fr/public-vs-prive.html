public vs private

La différence enfin claire — expliquée avec ta propre classe Contact du PhoneBook.

défile
01

La réponse en 10 secondes

+ 🔓

public

accessible depuis partout

Tout le monde peut l'utiliser : ton main, une autre classe, n'importe quel code à l'extérieur. C'est la porte d'entrée de ta classe.

− 🔒

private

intérieur de la classe uniquement

Seules les méthodes de la classe elle-même y ont accès. Depuis l'extérieur (ton main), c'est interdit — le compilateur refuse.

💡

La vraie question n'est pas « qui » a le droit, mais « OÙ est écrit le code ? ». Le même attribut est accessible depuis l'intérieur de la classe et interdit depuis l'extérieur.

02

L'analogie de la maison 🏠

Imagine qu'une classe = une maison. Il y a ce qui se passe à l'intérieur, et ce qui est visible depuis la rue.

🔒 private · l'intérieur
  • 🛋️Ton salon, ta chambre, ton journal intime. Personne dans la rue n'y touche directement.
  • 👨‍👩‍👧Seuls les habitants (les méthodes de la classe) peuvent y accéder et les modifier.
  • 🤫C'est exactement ton _darkestSecret : un secret, donc caché.
🔓 public · la façade
  • 🔔La sonnette, la boîte aux lettres, la porte. Tout le monde dans la rue peut les utiliser.
  • 🚪Ce sont tes méthodes publiques : setFirstName(), getFirstName()…
  • 🤝Le seul moyen « propre » pour l'extérieur d'interagir avec l'intérieur.
03

Qui peut accéder à quoi ?

D'où tu écris le code… accès à public accès à private
Depuis l'intérieur de la classeex: dans Contact.cpp, une méthode de Contact ✓ OUI ✓ OUI
Depuis l'extérieurex: dans ton main(), via un objet Contact ✓ OUI ✗ NON
Depuis une classe enfant (héritage)tu verras ça en CPP02/03 — protected ✓ OUI ✗ NON
04

Ta classe Contact, décortiquée

Voici ton vrai Contact.hpp. Regarde bien les deux blocs : les données privées en haut, l'interface publique en bas.

Contact.hpp
class Contact
{
private:                          // 🔒 LES DONNÉES — cachées du monde
    std::string _firstName;
    std::string _lastName;
    std::string _nickname;
    std::string _phoneNumber;
    std::string _darkestSecret;   // un "secret"... donc privé, logique !

public:                           // 🔓 L'INTERFACE — la porte d'entrée
    void        setFirstName(std::string str);  // pour ÉCRIRE
    std::string getFirstName(void);             // pour LIRE
    // ... idem pour lastName, nickname, etc.
};

Et dans Contact.cpp, le setter touche à l'attribut privé… et c'est autorisé, parce qu'on est à l'intérieur de la classe :

Contact.cpp
void Contact::setFirstName(std::string str)
{
    this->_firstName = str;   // ✅ OK : on est DANS la classe Contact
}

06

Ce qui se passe si tu triches

Si depuis ton main tu essaies d'accéder directement à un attribut privé, le code ne compile même pas. Ce n'est pas un crash à l'exécution : c'est le compilateur qui te bloque tout de suite.

main.cpp
Contact contact;

contact.setFirstName("Bob");        // ✅ OK   → setFirstName est public
std::cout << contact.getFirstName(); // ✅ OK   → getFirstName est public

contact._firstName = "Bob";          // ❌ ERREUR → _firstName est private
std::cout << contact._darkestSecret; // ❌ ERREUR → _darkestSecret est private
terminal — c++ -Wall -Wextra -Werror
error: 'std::string Contact::_firstName' is private within this context
    contact._firstName = "Bob";
            ^~~~~~~~~~
note: declared private here
    std::string _firstName;
                ^~~~~~~~~~

👉 La solution : passer par la porte d'entrée publique, donc contact.setFirstName("Bob").

07

Pourquoi s'embêter ? L'encapsulation

« Pourquoi cacher mes variables alors que je pourrais tout mettre en public ? » Voici les vraies raisons — c'est le cœur de la POO.

// 01

🛡️ Protéger les données

Un setter peut vérifier avant d'écrire (ex : refuser un numéro de téléphone vide). En public direct, n'importe qui peut mettre n'importe quoi.

// 02

🔧 Liberté de changer l'intérieur

Tant que tes méthodes publiques gardent le même nom, tu peux tout refaire à l'intérieur sans casser le code qui utilise ta classe.

// 03

🎯 Une interface claire

Celui qui utilise Contact voit seulement set/get. Il n'a pas à savoir comment c'est rangé dedans. Simple à utiliser.

// 04

🚫 Éviter les états invalides

Tu contrôles exactement comment ton objet peut être modifié. Moins de surprises, moins de bugs « impossibles ».

08

Le bonus que personne ne te dit

// struct vs class

En C++, la seule différence entre struct et class est l'accès par défaut : class → tout est private par défaut, struct → tout est public par défaut. C'est tout !

// l'underscore _

Le _ devant _firstName n'est pas obligatoire en C++. C'est une convention (42 et beaucoup d'autres) pour repérer d'un coup d'œil un attribut privé.

// vérifié à la compilation

private est une barrière du compilateur, pas une sécurité à l'exécution. Si tu violes l'accès, ça ne compile pas — ça ne « plante » pas plus tard.

// c'est par classe, pas par objet

Une méthode de Contact peut lire les attributs privés d'un autre Contact (utile pour le constructeur de copie). La barrière protège de l'extérieur de la classe, pas entre objets d'une même classe.

// et protected ? (à venir)

Il existe un 3ᵉ mot : protected. C'est comme private, mais les classes enfants (héritage) y ont accès aussi. Tu le rencontreras en CPP02/03.

📌 À retenir

  • →public = accessible de partout (ton main, autres classes).
  • →private = accessible uniquement depuis l'intérieur de la classe.
  • →La règle d'or : « d'où est écrit le code ? » — pas « qui ».
  • →On met les données en private et on expose des méthodes en public (les getters/setters) = encapsulation.
  • →Violer private depuis l'extérieur = erreur de compilation, jamais un crash silencieux.
  • →Ton PhoneBook fait exactement ça : Contact cache ses champs, le reste passe par set/get.
