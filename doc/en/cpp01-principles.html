02

Stack vs Heap — the big one

The central question of ex00: where does your object live, and who destroys it? This is the memory decision in C++.

📚

The Stack

  • ⚡Fast, automatic. Zombie z("x");
  • 🧹Destroyed by itself at the end of the { } (scope)
  • ⏱️Lifetime = limited to the current block
  • ✅No leak possible — you manage nothing
🗃️

The Heap

  • 🔑Manual. new Zombie("x") → returns a pointer
  • ♾️Survives beyond the function that created it
  • 🧨You MUST delete it — otherwise memory leak
  • 🎯Use it when the object must live longer
🧭

The decision rule: must the object outlive the function that creates it? Yes → heap (new). No → stack. That's exactly newZombie (heap, we return it) vs randomChump (stack, local use).

03

new / delete (and arrays)

C++ replaces malloc/free (forbidden here) with new/delete, which call the constructor and the destructor. For an array, ex01 asks for a single allocation: new Zombie[N].

zombieHorde.cpp
Zombie* zombieHorde( int N, std::string name )
{
    if (N <= 0)
        return NULL;

    Zombie* horde = new Zombie[N];   // ONE single allocation for N objects

    for (int i = 0; i < N; i++)
        horde[i].setName(name);
    return horde;                  // pointer to the 1st zombie
}
⚠️

The classic trap: what you allocate with new[] is freed with delete[] (with the brackets!), not delete. Otherwise it's undefined behaviour. → delete [] horde;

04

Pointer vs Reference

The heart of the module (ex02). A reference is just another name (an alias) for a variable that already exists. The proof: in ex02, the variable, the pointer and the reference all share the same address.

CriterionPointer  std::string*Reference  std::string&
can be null?yes (NULL)no — always valid
must be initialized?no (dangerous)yes — at declaration
can "point" elsewhere later?yes — reassignableno — bound for life
access syntaxptr->x / *ptrref.x (like the variable)
you store…an addressan alias
ex02/main.cpp
std::string  brain = "HI THIS IS BRAIN";
std::string* stringPTR = &brain;   // a pointer: stores the ADDRESS of brain
std::string& stringREF = brain;    // a reference: ANOTHER NAME for brain

// &brain == stringPTR == &stringREF  → the SAME address
// brain  == *stringPTR == stringREF   → the SAME value
05

Member reference vs member pointer

ex03 makes you choose. Two almost identical humans, but one design difference that dictates everything:

🗡️

HumanA → reference

  • •Always armed, receives the weapon in the constructor
  • •Never changes weapon, never without one
  • →Member Weapon& _weapon; (never null)
🛡️

HumanB → pointer

  • •May not have a weapon (at first)
  • •The weapon is added later via setWeapon()
  • →Member Weapon* _weapon; (can be NULL)
🧠

The lesson: "something can be absent or change" → pointer. "always present, fixed at construction" → reference. The code tells the intent.

06

Pointers to member functions

ex05 forces you to avoid the "forest of if/else". The solution: an array of pointers to member functions. Strange syntax, but logical:

Harl.cpp — complain()
std::string levels[4] = {"DEBUG", "INFO", "WARNING", "ERROR"};

// a pointer to a member function of Harl, taking nothing, returning void:
void (Harl::*funcs[4])( void ) = {
    &Harl::debug, &Harl::info, &Harl::warning, &Harl::error
};

for (int i = 0; i < 4; i++)
    if (levels[i] == level)
        (this->*funcs[i])();   // ← we call through the pointer

How to read it: void (Harl::*funcs[4])(void) = "an array of 4 pointers to methods of Harl that take void and return void". You call them with (this->*funcs[i])().

The switch and its fall-through

ex06 exploits a behaviour often seen as a bug: without a break, a case falls into the next one. Here it's intentional: "this level and above". Click a level 👇

Simulator for ./harlFilter <LEVEL> — watch which messages light up based on the chosen level.

ex06/main.cpp
switch (getLevel(argv[1]))
{
    case 0: harl.debug();   // no break → falls into the next case
    case 1: harl.info();
    case 2: harl.warning();
    case 3: harl.error(); break;
    default: std::cout << "[ Probably complaining... ]\n";
}
08

C++ file streams

ex04 forbids C functions (fopen, read…) and std::string::replace. You use the streams: std::ifstream (read), std::ofstream (write), std::stringstream (load everything into memory), and you replace by hand with find + substr.

FileReplacer.cpp — the idea
std::ifstream  inFile(_filename.c_str());   // open for reading
if (!inFile.is_open()) /* handle the error */;

std::stringstream buffer;
buffer << inFile.rdbuf();                // swallow the whole file at once
std::string content = buffer.str();

// replace s1 with s2 WITHOUT std::string::replace :
while ((found = content.find(_s1, pos)) != std::string::npos) {
    result += content.substr(pos, found - pos);
    result += _s2;
    pos = found + _s1.length();
}

📌 Key takeaways (eval defense)

  • →Stack = automatic, destroyed at the end of the scope. Heap (new) = manual, survives, must be deleted.
  • →new ↔ delete, new[] ↔ delete[]. Never mix, never forget (= leak).
  • →A reference = an alias: never null, init mandatory, not reassignable. A pointer = an address: can be null and change.
  • →Member reference = "always there" (HumanA). Member pointer = "may be missing/change" (HumanB).
  • →Pointer to member function: void (Harl::*f)(void), call (this->*f)() → replaces the if/else.
  • →switch without break = fall-through (intentional in ex06 for "this level and above").
  • →C++ I/O = streams (ifstream/ofstream), not the C functions.
