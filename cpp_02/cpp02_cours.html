01

La Forme Canonique Orthodoxe (OCF)

À partir de ce module, toute classe doit implémenter 4 fonctions membres. C'est le cœur de tout le module.

FonctionSignature typeQuand appelée
Constructeur par défautClassName();ClassName a;
Constructeur par copieClassName(const ClassName& o);ClassName b(a);
Opérateur d'affectationClassName& operator=(const ClassName& o);c = b;
Destructeur~ClassName();fin de portée / delete

Pourquoi 4 fonctions ?

Parce que le compilateur les génère automatiquement si tu ne les écris pas. Le but de l'OCF est de te forcer à les écrire toi-même pour contrôler ce qui se passe (et, plus tard, gérer la mémoire allouée). Ici on les écrit surtout pour afficher des messages et comprendre quand chacune est appelée.

🔑
La distinction qui piège tout le monde :
Copie vs affectation. Fixed b(a); → constructeur de copie (b n'existe pas encore, on le crée à partir de a). c = b; → opérateur d'affectation (c existe déjà, on écrase sa valeur).

L'astuce mentale : "naissance / vie / mort"

Un objet naît par un constructeur (défaut, copie, ou un autre), vit, puis meurt par le destructeur. Si tu traces les messages dans la sortie de l'ex00, compte les "Default/Copy constructor" → il doit y avoir exactement autant de "Destructor".

Dans Fixed c; c = b; — combien de constructeurs sont appelés pour c ?
Indice : c est déjà construit par défaut à la ligne d'avant. La ligne c = b n'appelle PAS de constructeur, juste operator=.

Pourquoi operator= retourne ClassName& ?

Pour permettre le chaînage : a = b = c;. On retourne *this. Pense aussi à te protéger de l'auto-affectation (a = a;) avec if (this != &other) — pas critique ici, mais bonne habitude pour les modules suivants.

getRawBits() et setRawBits() ex00

Ce sont les deux seules méthodes d'accès à l'entier interne _value. L'interface obligatoire de l'ex00 :

int  getRawBits( void ) const;   // retourne _value tel quel (l'entier brut)
void setRawBits( int const raw );  // écrit _value directement
🔑
getRawBits() est const parce qu'il ne modifie pas l'objet. setRawBits() ne l'est pas car il écrit _value. Penser à marquer const toutes les méthodes qui ne mutent pas l'objet — sinon le compilateur refuse de les appeler sur un objet const.

Ces méthodes sont aussi utiles en interne : dans operator= et le constructeur de copie, tu peux copier le raw directement via this->setRawBits(other.getRawBits()) sans te soucier de l'échelle.

02

Surcharge d'opérateurs (polymorphisme ad-hoc)

Le "polymorphisme ad-hoc", c'est juste : un même symbole (+, <, <<…) qui se comporte différemment selon les types des opérandes.

En C++ tu peux redéfinir ce que font les opérateurs sur tes types. a + b n'est qu'un appel déguisé à une fonction nommée operator+.

// Ces deux lignes sont équivalentes :
Fixed r = a + b;
Fixed r = a.operator+(b);   // version "membre"

Deux façons de surcharger

FormeExempleNote
MembreFixed operator+(const Fixed&) const;l'opérande gauche = *this
Libre (non-membre)std::ostream& operator<<(std::ostream&, const Fixed&);nécessaire quand l'opérande gauche n'est pas ta classe
⚠️
Le mot-clé friend est interdit dans ce module (note de bas de page du sujet → sinon -42). Donc pour operator<<, tu fais une fonction libre qui appelle des méthodes publiques de ta classe (genre toFloat()). Pas besoin de friend.

Comparaison, arithmétique ex02

Une fois que tu stockes la valeur en interne (le "raw"), tous les opérateurs comparent/calculent sur cette valeur entière. Le truc malin : tu peux écrire != en réutilisant ==, >= à partir de <, etc. Moins de code = moins de bugs. À toi de trouver lesquels dérivent des autres.

Incrément / décrément : le piège pré vs post

C'est LA subtilité de l'ex02. Le compilateur distingue les deux par un paramètre int factice :

Fixed& operator++();        // PRÉ-incrément : ++a   (retourne référence)
Fixed  operator++(int);     // POST-incrément : a++  (le (int) est bidon, jamais utilisé)
  • Pré (++a) : modifie puis retourne l'objet → retourne une référence.
  • Post (a++) : doit retourner la valeur d'avant → tu fais une copie, tu incrémentes *this, tu retournes la copie (par valeur).
Pourquoi le post-incrément retourne par valeur et pas par référence ?
L'ancienne valeur est une variable locale qui disparaît à la fin de la fonction — retourner une référence dessus serait une référence vers du vide (dangling). Donc par valeur.
03

Les nombres à virgule fixe (fixed-point)

Le concept mathématique derrière toute la classe Fixed. Comprends ça et le reste coule.

Idée : stocker un nombre décimal dans un entier, en réservant un nombre fixe de bits pour la partie fractionnaire. Ici : _fractionalBits = 8 (constant, toujours 8).

La formule clé :

valeur réelle  =  raw / 2^8   =  raw / 256
raw           =  valeur réelle × 256

Le facteur d'échelle est 2^8 = 256. Tout passe par là.

Valeur réelleraw stockécalcul
000 × 256
12561 × 256
10256010 × 256
0.0039062511 / 256 = ε le plus petit
💡
Le fameux 0.00390625 dans la sortie de l'ex02 ? C'est 1/256 — la plus petite valeur représentable, le ε. Quand tu fais ++a sur un fixed à 0, le raw passe de 0 à 1, soit 0.00390625 en réel. Tout s'explique par 256.

Pourquoi décaler de bits plutôt que multiplier ?

Multiplier/diviser par 256, c'est décaler de 8 bits. x << 8 ≡ x * 256, et x >> 8 ≡ x / 256 (pour les entiers). C'est rapide et c'est pour ça que les fixed-point sont utilisés en graphisme/son. Mais attention : pour la multiplication de deux fixed entre eux, l'échelle se multiplie aussi (256×256) → il faudra réajuster. Réfléchis à comment.

04

Conversions int ↔ float ↔ fixed ex01

Quatre constructeurs/méthodes qui font le pont entre les types. Tout repose sur la formule du chapitre 03.

OpérationDirectionLogique (à toi de coder)
Constructeur depuis intint → rawl'entier est décalé pour entrer dans le format fixed
Constructeur depuis floatfloat → rawmultiplier par l'échelle puis arrondir (roundf)
toFloat()raw → floatdiviser le raw par l'échelle (en flottant)
toInt()raw → intdiviser le raw par l'échelle (en entier)
⚠️
Le piège du float : quand tu convertis un float, tu obtiens rarement un nombre rond après multiplication. roundf (autorisé, depuis <cmath>) sert à arrondir au plus proche au lieu de tronquer. C'est pourquoi 42.42f ressort en 42.4219 dans l'exemple : la précision de 8 bits ne capture pas exactement 42.42.

Attention aux types dans les calculs

Si tu multiplies un int par 256 et que le résultat dépasse, ou si tu divises sans caster en float, tu perds la partie décimale. Pose-toi toujours la question : "à ce moment-là, je suis en entier ou en flottant ?"

Pourquoi toInt() et toFloat() ne donnent pas la même chose pour un raw donné ?
Même échelle (256), mais raw/256 en int tronque (ex: 2/256 = 0), alors qu'en float ça donne 0.0078125.
05

Surcharger operator<< ex01

Permet de faire std::cout << monFixed; directement.

Pourquoi une fonction libre et pas membre ? Parce que l'opérande de gauche est std::cout (un std::ostream), pas ton objet. Tu ne peux pas ajouter une méthode à std::ostream. La signature :

std::ostream& operator<<(std::ostream& os, const Fixed& f);
  • On retourne os (la référence reçue) → pour chaîner cout << a << b << endl;
  • À l'intérieur : os << f.toFloat(); — on insère la représentation flottante. Pas besoin de friend car toFloat() est publique.
🔗
Le chaînage marche parce que chaque << retourne le flux, qui sert d'opérande gauche au << suivant. C'est exactement le même principe que a = b = c avec operator=.
06

Membres static : min / max ex02

Une fonction static appartient à la classe, pas à un objet précis.

On l'appelle avec Fixed::max(a, b) — remarque le :: et l'absence d'instance devant. Une méthode static n'a pas de this, donc elle ne peut accéder qu'à ses paramètres (et aux membres static comme _fractionalBits).

Pourquoi 4 versions (2 min, 2 max) ?

Le sujet demande une version avec des références non-const et une avec des références const :

static Fixed&       max(Fixed& a, Fixed& b);
static const Fixed& max(const Fixed& a, const Fixed& b);

La const-correctness : si tu passes des objets const, seul l'overload const peut les accepter (et il renvoie du const, pour que tu ne puisses pas modifier un const via le retour). C'est de la surcharge sur la const-ness — encore du polymorphisme ad-hoc.

⚠️
Retourner une référence ici est volontaire : on renvoie l'un des deux objets existants passés en paramètre, pas une copie. Comme ils existent en dehors de la fonction, la référence reste valide. (Contrairement au post-incrément où l'objet était local !)
07

BSP : point dans un triangle ex03

Optionnel ("you can pass this module without completing exercise 03"), mais joli. Concept géométrique seulement.

L'idée mathématique courante : un point est dans le triangle ABC si on calcule l'aire signée des 3 sous-triangles formés avec le point (PAB, PBC, PCA), et qu'elles ont toutes le même signe que l'aire totale.

Aire signée d'un triangle (P1, P2, P3) :

aire = ( (x1-x3)(y2-y3) - (x2-x3)(y1-y3) ) / 2
  • Calcule l'aire de ABC, puis des 3 sous-triangles avec point.
  • Le point est strictement dedans si les 3 sous-aires sont du même signe ET aucune n'est nulle.
  • Une sous-aire nulle = le point est sur une arête ou un sommet → le sujet demande false dans ce cas.
🧮
Tous ces calculs se font avec ta classe Fixed et ses opérateurs surchargés — c'est l'intérêt de l'exo : réutiliser ce que tu as construit. Les attributs x et y de Point sont const, ce qui a une conséquence directe sur ton operator= de Point — réfléchis à laquelle.

Pièges & checklist de défense

Erreurs classiques qui coûtent la note

  • Implémentation dans le .hpp → 0 à l'exo (sauf templates). Déclaration en .hpp, code en .cpp.
  • Include guards manquants → 0. Mets #ifndef / #define / #endif dans chaque header.
  • using namespace ou friend → -42. Interdits ici.
  • STL / containers / <algorithm> → -42. Pas avant les modules 08-09.
  • Fuites mémoire si tu utilises new (pas nécessaire ici, mais surveille).
  • Confondre copie et affectation dans les messages → la sortie ne correspondra pas.
  • Oublier const sur getRawBits() et les méthodes qui ne modifient pas l'objet.
